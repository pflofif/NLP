
# Dataset Card

**Назва проєкту:** Система пошуку туристичних локацій Львова на основі відгуків користувачів.

**Задача:** Retrieval (Напрямок C).

- **Input:** Текстовий запит користувача природною мовою (пошук за стравою, атмосферою, ціною або сервісом).
- **Output:** Список (Топ-N) релевантних відгуків, які допомагають користувачу прийняти рішення.

**Джерело даних:** Реальні відгуки користувачів з платформи Google Maps для закладів громадського харчування Львова. Дані зібрані за допомогою open-source інструментів парсингу (https://github.com/omkarcloud/google-maps-scraper?tab=readme-ov-file).

**Обсяг:** 

- 6000+ унікальних документів (відгуків) після очищення.
- 40 запитів (queries) для тестування системи.

**Мова/домени:** UA. Домен: Туризм, гастрономія, сфера послуг (HoReCa).

**Очищення:**

- Видалено зайві пробіли та символи переносу рядка.
- Уніфіковано використання апострофа (замінено ' та ` на ’).
- Анонімізовано дані: замінено прямі посилання (URL), номери телефонів та email на відповідні теги-заглушки: `<URL>`, `<PHONE>`, `<EMAIL>`.
- Видалено порожні записи та тексти без змістовного навантаження.

**Ризики:**

- **Шум:** Наявність відгуків, що складаються лише з емодзі або вигуків ("Клас!", "Супер").
- **Суб'єктивність:** Один і той самий заклад може мати діаметрально протилежні оцінки.
- **Мова:** Використання діалектизмів ("зупа", "філіжанка") або суржику, що може ускладнити класичний пошук за ключовими словами.
- **Дублі:** Існує ризик наявності ідентичних відгуків від одного користувача для різних філій закладу.

## Splits & leakage (ЛР5)

- **Стратегія:** Group split по place_name, seed=42, 80/10/10 (train/val/test)
- **Розміри:** train=3723, val=444, test=473
- **Group leakage:** 0 — заклади повністю ізольовані між сплітами
- **Exact duplicates train∩test:** 7 (однакові тексти з різних закладів — мінімальний ризик)
- **Near-duplicates (cosine≥0.95):** 226 пар (переважно короткі відгуки)
- **Template leakage:** 0
- Manifest: docs/splits_manifest_lab5.json

## Classification baseline (ЛР6)

- **Напрям:** C (Retrieval) — бінарна класифікація пар (query, doc) → relevant / not_relevant
- **Запити:** 40, релевантних пар: 43 (labels.csv)
- **Побудова датасету:** 1–2 позитиви + 20 random-negatives на запит; ознака = query+doc конкатенація
- **Split:** query-level 80/10/10, seed=42 (гарантує позитиви в кожному сплті)
- **B1:** text_v2, TF-IDF unigrams, LogReg — val acc=0.9524, val F1=0.4878
- **B2 (winner):** noun_adj_text, TF-IDF bigrams, balanced — val acc=0.9881, val F1=0.9255
- **Ризики:** малий тест (4 запити, 3 позитиви); hard negatives потрібні для кращого розмежування

## Linear SVM + char-ngrams (ЛР7)

- **Дисбаланс класів:** Так, ~19:1 (негативних до позитивних) у train. Дисбаланс підтверджений у всіх сплітах.
- **Noisy text / translit / spelling variation:** Мінімально — корпус переважно стандартна українська письмова мова. Діалектизми ("зупа", "філіжанка") присутні, але не домінують.
- **Чи допомогли char-ngrams:** Ні — на validation val F1 збігається (0.9255), але на test char-only (V3a) дає F1=0.6137, word+char (V3b) дає F1=0.7566. Обидва гірші за word-only (F1=1.0). Причина: чистий текст без значного translit/slangu не виграє від char-представлення.
- **Найпроблемніший клас:** `relevant` (minority) — підтримка всього 3–5 прикладів у val/test. Будь-яка помилка на цьому класі сильно впливає на macro-F1.
- **Кращий класифікатор:** LinearSVC word(1,2) = LogReg word(1,2) — обидва досягають Test F1=1.0 при достатньому сигналі (noun_adj_text).
- **Поріг рішення:** CHOSEN_THR ≈ −0.768 (підібраний за max F1 на val). При default порозі (0.0) recall=0.75 (один позитив втрачається). Логіка: recall-first для IR-задачі, де FN дорожчий за FP.
- **Компроміс precision/recall:** При порозі −0.768: P=1.0, R=1.0, F1=1.0 на val. При порозі 0.0: P=1.0, R=0.75, F1=0.857. Вибір порогу нижче default відновлює всі позитиви без жертви precision.
