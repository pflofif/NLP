# 1. Побудова датасета пар

| Компонент          | Деталь                                               |
| --------------------------- | ---------------------------------------------------------- |
| Позитивні пари | з `labels.csv` (43 пар)                              |
| Негативні пари | 20 random-sample на запит                           |
| Ознака                | `query_text + " " + doc_text` (конкатенація) |
| Split                       | Query-level 80/10/10, seed=42                              |

Split за запитами гарантує, що кожен з трьох підмножин містить позитивні приклади
(на відміну від doc-level split, де всі 43 позитивних можуть опинитись у train).

---

## 2. Базові моделі

| Baseline | Текст        | N-грами | class_weight |
| -------- | ----------------- | ------------ | ------------ |
| B1       | `text_v2`       | (1,1)        | —           |
| B2       | `noun_adj_text` | (1,2)        | balanced     |

Обидва: TF-IDF (`sublinear_tf=True`, `max_features=30 000`) + LogisticRegression (lbfgs, max_iter=500).

---

## 3. Результати

| Baseline                                  | Val Acc          | Val macro-F1     | Test Acc        | Test macro-F1   |
| ----------------------------------------- | ---------------- | ---------------- | --------------- | --------------- |
| B1: text_v2, unigrams                     | 0.9524           | 0.4878           | 0.9880          | 0.8969          |
| **B2: noun_adj, bigrams, balanced** | **0.9881** | **0.9255** | **1.000** | **1.000** |

**Переможець: B2** — вагомий результат за val macro-F1 (0.9255).

Примітка: test-set містить 4 запити (3 позитиви, 80 негативів); val-set — 4 запити.
Невеликий розмір тестової вибірки зумовлений малою кількістю анотованих запитів (40).

---

## 4. Аналіз помилок

На тестовій виборці B2 не допустив жодної помилки (TN=80, FP=0, FN=0, TP=3).
На валідаційній виборці B1 не класифікував релевантні документи (`relevant` F1 ≈ 0),
тоді як B2 досяг F1=0.9255 завдяки bigrams і балансуванню класів.

Типові причини помилок на val у B1:

- **FP**: лексичний overlap (схожа лексика, але інший заклад)
- **FN**: специфічна термінологія запиту не зустрічається в train-корпусі

Файл з прикладами помилок: `tests/error_cases_lab6.jsonl`

## 5. Ключові ознаки B2 (relevant)

`надійка приємний`, `дитинство ціна`, `атмосфера топ`, `топ стендап`, `дитинство`

## 6. Що можна покращити

- **Hard negative mining** — відбирати важкі негативи (BM25 top-k, але не в labels.csv)
- **Threshold tuning** — оптимізувати поріг на val для zменшення FP
- **BM25 / dense retrieval** — порівняти з TF-IDF на метриці MAP@10 або NDCG
